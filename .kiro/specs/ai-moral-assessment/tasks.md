  - `npm run dev`（Vite dev server）・`npm run build`・`wrangler pages deploy` の各コマンドが動作することを確認する

- [x] 2. ゲームのコアデータ型を定義する

- [x] 2.1 (P) シナリオ・診断パラメータの型を定義する
  - シナリオ本文・選択肢・論破テキスト・6軸スコア重みを持つシナリオデータの型を定義する
  - 6軸（結果効率・規律・人情・自己保身・共感・論理）の数値型と、選択肢ごとの差分型を定義する
  - `Result<T, E>` を独自定義の直和型（ok/error フィールド）として定義し、以降の非同期処理で一貫して使用できるようにする
  - シナリオ読み込みエラーを「未発見・スキーマ不正・ネットワーク障害」の3種の判別可能な型として定義する
  - _Requirements: 6.2, 3.1_

- [x] 2.2 (P) 診断タイプ・ゲーム状態の型を定義する
  - 16タイプを識別する4文字コード型（功利/共感・秩序/自由・能動/受動・個人/集団の各二択）を定義する
  - タイプ名・キャッチフレーズ・説明文・ライバルタイプIDを持つ性格タイプ型を定義する
  - ゲーム進行フェーズ（読み込み中・シナリオ表示・二択・論破・結果・エラー）と、各フェーズの状態を持つゲーム状態型を定義する
  - GameContext から発行される全アクション（ゲーム開始・選択・論破終了・完了・エラー）を判別可能な型として定義する
  - _Requirements: 3.3, 3.6, 1.2_

- [ ] 3. 診断ビジネスロジックを実装する

- [ ] 3.1 6軸スコアの集計ロジックを実装する
  - 現在のスコアに選択肢の差分を加算して新しいスコアオブジェクトを返す純粋関数を実装する
  - 全軸ゼロの初期スコアを返す初期化関数を実装する
  - 副作用なし・同一入力で同一出力であることをユニットテストで確認する（境界値・負値・ゼロ値）
  - _Requirements: 3.1, 3.2, 3.4_

- [ ] 3.2 16タイプ判定ロジックを実装する
  - 全16タイプのコード・タイプ名・説明文・ライバルタイプをマップ定数として実装する（`docs/gdd/05-type-01-08.md`, `06-type-09-16.md` を参照）
  - 最終スコアから4つの二元軸を導出するロジックを実装する
    - 功利/共感: result_efficiency vs empathy_kindness
    - 秩序/自由: rule_discipline vs self_preservation
    - 能動/受動: (result_efficiency + rule_discipline + logic_coldness) vs (humanity_morality + self_preservation + empathy_kindness)
    - 個人/集団: humanity_morality vs logic_coldness
  - 同点タイブレークルール（U>E, O>F, A>P, I>C）を適用して必ず1タイプを返す関数を実装する
  - 全16タイプコードのパターンをユニットテストで網羅する
  - _Requirements: 3.2, 3.3, 3.5, 3.6_

- [ ] 3.3 (P) シェアテキストと結果URLの生成ロジックを実装する
  - タイプ名・ハッシュタグ（#AIモラル診断）・サービスURLを含む X/LINE 向けシェアテキストを生成する関数を実装する
  - 診断結果を TypeCode クエリパラメータに変換して結果共有URLを組み立てる関数を実装する
  - _Requirements: 5.2, 5.5, 4.5_

- [ ] 4. データ管理とセッション管理を実装する

- [ ] 4.1 (P) セッション状態の永続化機能を実装する
  - ゲーム進行状態（現在問・スコア・選択済み回答）を localStorage に保存・復元する機能を実装する
  - localStorage が利用できない場合（プライベートモード・容量超過）はエラーをキャッチし、インメモリで継続する
  - 今日の日付キーでデイリーモードのプレイ済み状態を記録・判定する機能を実装する
  - SessionManager 以外のコンポーネントが localStorage に直接アクセスしないよう、単一窓口として設計する
  - _Requirements: 1.6_

- [ ] 4.2 (P) シナリオの読み込みとスキーマ検証機能を実装する
  - `publishDate` フィールドで今日の日付に対応するシナリオを特定してデイリーシナリオを取得する機能を実装する
  - 取得したJSONをシナリオ型スキーマと照合し、不正データは `INVALID_SCHEMA` エラーとして扱う
  - ネットワークエラー・404 をそれぞれ適切なエラー型にマッピングして `Result` 型で返す
  - _Requirements: 6.1, 6.2, 6.4, 6.6_

- [ ] 4.3 (P) シナリオ JSON データを作成する
  - CASE 01〜03 は `docs/gdd/09-case-01.md`〜`11-case-03.md` の設計に従って JSON 化する
  - CASE 04〜10 は `docs/gdd/08-scenario-design.md` のテーマカテゴリ（監視社会・生命倫理・親権・緊急事態・環境）から新規作成する
  - 各シナリオの本文は200〜350字、各選択肢に論破テキストと6軸パラメータ差分を付与する
  - `publishDate` を各ケースに割り当て、デイリーモードで配信できる状態にする
  - _Requirements: 6.1, 6.3, 6.4_

- [ ] 4.4 (P) ビルド時のシナリオ JSON スキーマ検証を組み込む
  - Vite プラグインまたはビルド前スクリプトとして、`public/scenarios/` 配下の全 JSON をシナリオ型スキーマで自動検証する仕組みを実装する
  - スキーマ違反があればビルドエラーとして出力し、不正データがデプロイされるのを防ぐ
  - _Requirements: 6.5_

- [ ] 6. UIコンポーネントを実装する

- [ ] 6.1 (P) シナリオ提示・選択UIを実装する
  - シナリオのカテゴリタグ・タイトル・本文・イラスト領域（Phase 1 はプレースホルダー）を表示するシナリオカードを実装する
  - A/B の大きな二択ボタンを上下配置し、スワイプ操作でも選択できるインタラクションを実装する
  - 現在の問番号と全問数（例: 3 / 8）を進捗バーで表示する
  - 全コンポーネントはスマホ縦持ちで操作が完結するモバイルファーストレイアウトにする
  - _Requirements: 1.1, 1.3, 1.4, 1.5, 1.7_

- [ ] 6.2 (P) 論破演出画面を実装する
  - 選択後に「AIが分析中」のローディング演出を1〜2秒表示してから論破テキストのタイピングを開始する
  - 論破テキストを1文字20msのタイピング演出でフルに表示する（アンマウント時にタイマーをクリーンアップする）
  - タイピング完了前でも「次のケースへ」ボタンで即時スキップできるようにする
  - 「異議あり」「次のケースへ」のどちらを押しても次フェーズへ進む同一の動作をする
  - _Requirements: 2.1, 2.2, 2.3, 2.4, 2.5, 2.6_

- [ ] 6.3 (P) 6軸レーダーチャートを実装する
  - ライブラリ不使用の純粋 SVG で正六角形レーダーチャートを実装する（バンドルサイズ最小化のため）
  - 6軸のラベル（結果・効率 / ルール・規律 / 人情・道徳 / 自己保身 / 共感・優しさ / 論理・冷酷さ）を頂点付近に配置する
  - CSS アニメーション（stroke-dashoffset）でスコアが0から最終値まで描画されるアニメーションを実装する
  - _Requirements: 4.2_

- [ ] 6.4 (P) 診断結果表示・シェアUIを実装する
  - タイプ名・英名・キャッチコピー・タイプイラスト領域（Phase 1 はプレースホルダー）・詳細説明をまとめた結果レポートを実装する
  - 最も対立するタイプ（ライバルタイプ）を付加情報として表示する
  - X（Twitter）・LINE・URLコピーの3種のシェアボタンを実装し、タップで生成済みシェアテキストを渡す
  - _Requirements: 4.1, 4.3, 4.4, 5.1, 5.2, 5.5_

- [ ] 7.1 ランディングページを実装する

  - プレイ済みの場合はデイリー完了メッセージと結果リンクを表示する

  - 無効な TypeCode が渡された場合はエラーメッセージを表示してトップページへ誘導する
  - OGP メタタグ（og:title・og:description・og:image・og:url・twitter:card）を TypeCode に基づいて動的に設定する

- [ ] 8. OGP 画像生成 Worker を実装する

- [ ] 8.1 OGP 画像生成 Worker を実装する
  - `?type=01〜16` のクエリパラメータを受け取り、`workers-og`（Satori + resvg）でタイプ専用の OGP 画像（PNG 1200×630）を生成する
  - Workers KV からフォントを取得して Satori に渡し、日本語テキストを正しく描画する
  - WASM の二重初期化を防ぐ `isInit` フラグをグローバルスコープで管理する
  - タイプID が 01〜16 の範囲外の場合は HTTP 400 を返す入力バリデーションを実装する
  - `Cache-Control: public, max-age=86400` を設定して CDN に 24 時間キャッシュさせる
  - _Requirements: 5.3, 5.4, 5.6_

- [ ] 9.1 (P) ビジネスロジックのユニットテストを実装する
  - スコア加算が6軸すべてで正確に動作すること（境界値・負値・ゼロ）をテストする
  - 全16タイプのTypeCode判定が正確なこと、同点タイブレークが決定論的なことをテストする

  - シェアテキストがタイプ名・ハッシュタグ・URLを含むことをテストする
  - _Requirements: 3.2, 3.4, 3.5, 1.6, 5.2_

- [ ] 9.2 (P) GameContext の統合テストを実装する
  - ゲーム開始から選択・論破・完了までのフルフローで状態が正しく遷移することをテストする
  - ScenarioLoader が正常なJSONを返すケースと各エラーケースの動作をテストする
